## Objective

I have a major annoyance with vterm.

- When I switch away and back to a workspace, my vterm buffers for that workspace will re-order themselves into an apparantly random order

This is extremely annoying/disruptive.

I would like my vterm buffers to always remain in numbered order, and have the OPTION to move buffers around.

Can you help identify the cause and implement a fix

## Summary

### Root Cause
centaur-tabs rebuilds tabsets by prepending buffers, which reverses tab order on workspace switch. Three approaches failed before finding the fix:

1. **`centaur-tabs-force-update` in `persp-switch-hook`** — force-update rebuilds the buffer list but preserves existing tab positions within the tabset (only adds/removes, never re-sorts)
2. **Direct tabset manipulation in `persp-switch-hook`** — centaur-tabs rebuilds the tabset *after* the hook fires, overwriting our sort
3. **`:around` advice on `centaur-tabs-tabs`** — `centaur-tabs-tabs` is a `defsubst` (inlined at compile time), so advice on it has no effect

### Fix
Added `:after` advice on `centaur-tabs-buffer-update-groups` (`lg/centaur-tabs-sort-vterm-after-update`). This is the regular `defun` that rebuilds tabsets. After it runs, we iterate all tabsets via `centaur-tabs-tabsets` obarray and sort any containing vterm tabs by numeric index, modifying the symbol values directly — which is what the inlined `centaur-tabs-tabs` reads.

### Files Changed
- `doom.d/packages/centaur-tabs.el` — added `:after` advice on `centaur-tabs-buffer-update-groups`

### Key Decisions
- Sorting happens on every tabset rebuild, not just workspace switch — ensures consistency in all scenarios
- M-{/M-} manual reorder is overridden on next rebuild; to persist a custom order, use `SPC o r` (reindex) which changes buffer names/indices

### Gotchas
- `centaur-tabs-tabs` is a `defsubst` — cannot be advised. Must modify tabset symbol values directly.
- centaur-tabs rebuilds tabsets asynchronously relative to `persp-switch-hook` — hook-based sorting doesn't work. Advice on the rebuild function itself is the only reliable approach.
