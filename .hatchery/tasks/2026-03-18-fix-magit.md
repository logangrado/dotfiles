# Task: fix-magit

**Status**: complete
**Branch**: hatchery/fix-magit
**Created**: 2026-03-18 08:25

## Objective

After the previous commit added perspective-scoped magit buffer management, a bug
emerged: pressing `l a` (all logs) from within a log view showed commits from the
**wrong perspective's repository**.

## Context

With two perspectives A (project-a) and B (project-b) open, doing `spc g g` → `l l`
→ `l a` in perspective B would show project-a's history instead of project-b's.

## Summary

### Root Cause

`lg/magit-kill-stale-log-buffers` is `:before` advice on `magit-log-setup-buffer`.
It killed **all** matching log buffers — including the currently active one (`Log-B`).
After killing `Log-B`, Emacs's `save-current-buffer` restoration failed, switching
`current-buffer` to another buffer (project-a's log). Then `magit-log-setup-buffer`
used that corrupted context to find the wrong repo.

### Fix 1 — `lg/magit-kill-stale-log-buffers` (`doom.d/packages/magit.el`)

Added `(cur (current-buffer))` and a `(not (eq buf cur))` guard so the advice never
kills the buffer it is being called from. This keeps `Log-B` alive, preserves the
correct `current-buffer` context, and allows `magit-get-mode-buffer` to find the
right repo.

### Fix 2 — `lg/magit-log-branches` (`doom.d/packages/magit.el`)

Removed the leftover `"test message"` string (was being passed as the `locked` arg —
a debugging artifact) and the unused `'magit-log-mode` arg from the
`magit-log-setup-buffer` call.

### Key insight

`magit-log-current` (`l l`) and `magit-log-all` (`l a`) both call
`magit-log-setup-buffer` with `locked=nil`, so there is only ever **one** log buffer
per repo. `l l` → `l a` updates that buffer in-place; no stale buffer is created.
The advice only needs to clean up orphans from previous sessions, not the live buffer.
